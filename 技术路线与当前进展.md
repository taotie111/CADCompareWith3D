# 水利工程 CAD三维对比智能核查系统 技术方案

## 一、系统概述

### 1.1 项目背景

随着水利工程规模化发展，传统人工巡检方式已难以满足精细化管理需求。本系统旨在构建一个以CAD设计为基准，融合无人机倾斜摄影、三维重建与AI视觉分析的智能化核查平台，实现：

- **设计-施工一致性自动核查**：将CAD图纸与实际施工结果进行自动化几何比对
- **工程隐患智能识别**：通过AI视觉分析技术自动识别裂缝、变形、违规施工等问题
- **风险预警与辅助决策**：基于规则引擎和LLM智能复核生成风险等级和处置建议

**核心价值**：让“图纸自动去检查现实世界”，从根本上提升工程监管效率与质量。

### 1.2 业务场景

| 场景类型 | 典型对象 | 对比内容 |
|---------|---------|---------|
| 大坝工程 | 坝体、溢洪道、引水隧洞 | 轴线偏移、断面尺寸、坡度 |
| 水闸工程 | 闸室、闸墩、启闭机房 | 平面位置、高程、净空尺寸 |
| 渠道工程 | 渠身、衬砌、桥涵 | 断面尺寸、轴线偏差 |

### 1.3 约束条件

| 约束项 | 要求 |
|-------|------|
| 部署方式 | 私有化部署 |
| 输入格式 | DWG（主）、ITF/IFC（后续） |
| 坐标系 | CGCS2000 / WGS84 |
| 场景输入 | 网页地址 / 3D Tileset服务地址 |
| 鉴权方式 | Bearer Token |
| 测试策略 | 不落库测试验证 |

## 三、核心功能模块

### 3.1 CAD解析与语义重建

**功能描述**：将CAD图纸（DWG/DXF格式）解析为可计算的几何对象，并建立语义映射关系。

**处理流程**：

```
DWG/DXF文件 → 几何解析 → 图层识别 → 块解析 → 语义映射 → 标准化对象
```

**关键能力**：

| 能力 | 说明 |
|-----|------|
| 几何提取 | 线段、圆弧、多段线、块的坐标与属性 |
| 图层映射 | Layer Name → Object Type（坝体、闸墩、闸室等） |
| 拓扑重建 | 管线网络、结构关系的关联表达 |
| 坐标转换 | 本地坐标 → CGCS2000/WGS84 |

**输出格式**：

```json
{
  "objects": [
    {
      "id": "sluice_axis_01",
      "type": "sluice_axis",
      "location": "闸室1段",
      "x": 1000.0,
      "y": 2000.0,
      "z": 35.0,
      "length_m": 48.0,
      "width_m": 12.0,
      "height_m": 18.0,
      "source_type": "dwg",
      "confidence": 0.96,
      "trace": {
        "file": "design_main.dwg",
        "layer": "SLUICE_AXIS"
      }
    }
  ]
}
```

### 3.2 倾斜摄影与三维重建

**功能描述**：通过无人机倾斜摄影生成实景三维模型，为后续对比提供基准数据。

**处理流程**：

```
影像采集 → SfM/MVS重建 → Mesh生成 → DOM/DSM生产 → 3D Tiles切片 → 发布服务
```

**技术指标**：

| 指标 | 要求 |
|-----|------|
| 重建覆盖率 | ≥95% |
| 平面精度 | ≤5cm |
| 高程精度 | ≤10cm |
| 输出格式 | 3D Tiles / OSGB |

### 3.3 空间配准

**功能描述**：将设计模型与实景模型统一到同一坐标系下，确保可进行精确对比。

**配准策略**：

| 方法 | 精度 | 适用场景 |
|-----|------|---------|
| 控制点配准(GCP) | 高 | 有控制点成果 |
| 人工锚点对齐 | 中 | 无控制点，但有显著地物 |
| ICP自动配准 | 中-低 | 无控制点，自动特征匹配 |

**当前实现**：

- POC阶段采用已知坐标直接对齐
- 后续支持控制点(GCP)导入
- 可配置配准精度阈值

### 3.4 差异检测引擎

**功能描述**：计算设计模型与实景模型之间的几何差异，识别偏差超标对象。

#### 3.4.1 对象匹配

采用**多因素加权评分**策略：

```
S_total = 0.35×S_id + 0.20×S_type + 0.20×S_dist + 0.15×S_dim + 0.10×S_src
```

| 因素 | 说明 | 权重 |
|-----|------|------|
| S_id | ID精确匹配 | 0.35 |
| S_type | 对象类型一致性 | 0.20 |
| S_dist | 空间距离得分 | 0.20 |
| S_dim | 尺寸相似度 | 0.15 |
| S_src | 数据来源权重(DWG=1.0, PDF=0.7) | 0.10 |

#### 3.4.2 几何指标计算

| 指标 | 计算公式 | 单位 |
|-----|---------|------|
| 平面偏移 | √((x₁-x₂)²+(y₁-y₂)²) | 米(m) |
| 高程偏差 | |z₁-z₂| | 米(m) |
| 尺寸偏差率 | |L₁-L₂|/|L₁| | 百分比(%) |
| 变形量 | 连续点云面法向偏差 | 米(m) |

#### 3.4.3 风险规则判定

| 规则ID | 规则名称 | 触发条件 |
|-------|---------|---------|
| GEO_PLANAR_OFFSET | 关键构件平面偏移 | 偏移值>阈值 |
| GEO_ELEVATION_DEVIATION | 关键构件高程偏差 | 高程差>阈值 |
| DIMENSION_DEVIATION_RATE | 关键尺寸偏差率 | 偏差率>阈值 |
| SEM_MISSING_CONSTRUCTION | 漏建检测 | 设计有→实景无 |
| SEM_UNPLANNED_CONSTRUCTION | 疑似违建 | 实景有→设计无 |
| STR_LOCAL_DEFORMATION | 局部变形疑似 | 变形量+面积超限 |

**阈值配置**（可动态调整）：

| 规则 | Low | Medium | High | Critical |
|------|-----|--------|------|----------|
| 平面偏移(m) | 0.05 | 0.10 | 0.20 | 0.30 |
| 高程偏差(m) | 0.03 | 0.08 | 0.15 | 0.25 |
| 尺寸偏差率(%) | 0.5 | 1.0 | 2.0 | 3.0 |

### 3.5 LLM智能复核

**功能描述**：利用大语言模型对风险事件进行智能复核，增强语义理解与解释能力。

#### 3.5.1 触发条件

| 条件 | 说明 |
|-----|------|
| 低匹配置信度 | 匹配得分<0.65 |
| PDF来源且临界 | PDF数据且指标在阈值边界±2cm |
| 规则冲突 | 同一对象触发多条规则 |
| 强制复核 | 配置review_mode=always |

#### 3.5.2 LLM输出Schema

```json
{
  "risk_interpretation": "闸室1段平面偏移0.234m，超出高风险阈值0.2m...",
  "evidence_refs": [
    {"type": "design_document", "id": "sluice_axis_01", "source_file": "design_main.dwg"}
  ],
  "confidence_delta": 0.02,
  "recommended_action": {
    "immediate": "立即组织现场复核测量",
    "short_term": "如确认超限，制定纠偏方案",
    "long_term": "加强施工过程轴线控制"
  },
  "schema_valid": true
}
```

#### 3.5.3 可靠性保障

- **强制JSON输出**：配置response_format为json_object
- **失败回退**：调用失败自动切换本地规则引擎结论
- **证据回指**：每条结论必须引用原始数据来源

## 四、数据模型设计

### 4.1 核心表结构

#### 4.1.1 项目与场景

```sql
CREATE TABLE projects (
  id CHAR(36) PRIMARY KEY,
  code VARCHAR(64) UNIQUE NOT NULL,
  name VARCHAR(255) NOT NULL,
  coordinate_system VARCHAR(64) DEFAULT 'CGCS2000',
  status VARCHAR(32) DEFAULT 'active',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE scenes (
  id CHAR(36) PRIMARY KEY,
  project_id CHAR(36) REFERENCES projects(id),
  scene_name VARCHAR(255) NOT NULL,
  capture_time DATETIME,
  version_no INT DEFAULT 1,
  status VARCHAR(32) DEFAULT 'draft'
);
```

#### 4.1.2 设计对象与实景对象

```sql
CREATE TABLE design_objects (
  id CHAR(36) PRIMARY KEY,
  project_id CHAR(36),
  scene_id CHAR(36),
  object_code VARCHAR(128),
  object_type VARCHAR(64) NOT NULL,
  layer_name VARCHAR(128),
  geom GEOMETRY(POINT, 4490),
  elevation_m DECIMAL(10,3),
  source_type VARCHAR(16) DEFAULT 'dwg',
  source_confidence DECIMAL(5,4),
  trace_json JSON
);
```

#### 4.1.3 差异结果与风险事件

```sql
CREATE TABLE diff_results (
  id CHAR(36) PRIMARY KEY,
  project_id CHAR(36),
  scene_id CHAR(36),
  diff_type VARCHAR(32),
  metric_name VARCHAR(64),
  metric_value DECIMAL(12,5),
  risk_level VARCHAR(16),
  status VARCHAR(32) DEFAULT 'new',
  match_confidence DECIMAL(5,4),
  manual_review_required TINYINT DEFAULT 0
);

CREATE TABLE risk_events (
  id CHAR(36) PRIMARY KEY,
  project_id CHAR(36),
  scene_id CHAR(36),
  rule_id VARCHAR(128),
  risk_type VARCHAR(128),
  level VARCHAR(16),
  title VARCHAR(255),
  description TEXT,
  suggestion TEXT,
  location GEOMETRY(POINT, 4490),
  llm_reviewed TINYINT DEFAULT 0,
  manual_review_required TINYINT DEFAULT 0,
  status VARCHAR(32) DEFAULT 'open'
);
```

### 4.2 数据流关系

```
projects → scenes → assets → models → design_objects
                ↓
          reality_data → ai_detections
                ↓
          registration_results → diff_results → risk_events → work_orders
                                      ↓
                               llm_calls (审计)
```
